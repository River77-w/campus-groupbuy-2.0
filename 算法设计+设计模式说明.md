### 1. AA 费用均摊算法

#### 1.1 算法概述

AA 费用计算是本系统的核心业务算法，用于将拼单总金额按目标人数均摊，得出每人应付金额。

#### 1.2 流程图

```
┌──────────────────────┐
│     开始：输入        │
│  totalAmount（总金额）│
│  participantCount（人数）│
└──────────┬───────────┘
           ▼
    ┌──────────────┐
    │ 参数合法性检查 │
    │ amount>0 且   │
    │ count>0 ?     │
    └───┬──────┬───┘
        │ YES  │ NO
        ▼      ▼
  ┌──────────┐  ┌───────────┐
  │ aa =     │  │ return 0  │
  │ amount / │  │ 直接返回  │
  │ count    │  └───────────┘
  └────┬─────┘
       ▼
  ┌──────────────┐
  │ 保留两位小数  │
  │ formatPrice()│
  │ parseFloat   │
  │ .toFixed(2)  │
  └──────┬───────┘
         ▼
  ┌─────────────┐
  │  返回 AA 值  │
  └─────────────┘
```

#### 1.3 伪代码

```
FUNCTION calculateAA(totalAmount, participantCount):
    IF totalAmount IS NULL OR participantCount IS NULL OR participantCount == 0:
        RETURN 0
    END IF

    aaAmount = totalAmount / participantCount
    RETURN formatPrice(aaAmount)    // 保留两位小数
END FUNCTION

FUNCTION formatPrice(price):
    RETURN parseFloat(price).toFixed(2)
END FUNCTION
```

#### 1.5 算法分析

| 维度 | 说明 |
|------|------|
| **时间复杂度** | O(1)，仅需一次除法运算 |
| **空间复杂度** | O(1)，无额外数据结构 |
| **边界处理** | 金额为 0、人数为 0 或未定义时返回 0，防止 NaN |
| **精度保证** | 使用 `toFixed(2)` 保留两位小数，避免 JavaScript 浮点数误差 |

### 2. 倒计时实时更新算法

#### 2.1 算法概述

倒计时算法负责计算拼单剩余时间，并根据剩余时长动态调整显示样式。算法分为三个层级：**核心计算函数** → **单卡片更新** → **全局轮询调度**。

FUNCTION getCountdown(endTime):
    now = new Date().getTime()
    end = new Date(endTime).getTime()
    diff = end - now                    // 毫秒

    IF diff <= 0:
        RETURN { text: "已结束", isUrgent: false, isEnded: true }

    minutes = floor((diff % (1000 * 60 * 60)) / (1000 * 60))

    IF hours > 0:

    ELSE IF minutes > 0:
        RETURN { text: "剩余 ${minutes}分${seconds}秒", isUrgent: true, isEnded: false }

FUNCTION updateListCountdowns():
    groupBuys = getAllGroupBuys()
    FOR EACH gb IN groupBuys:
        countdownEl = DOM.getElementById("countdown-" + gb.id)
        IF countdownEl IS NULL: CONTINUE

        countdown = getCountdown(gb.deadline)

        // 更新倒计时文本
        countdownEl.querySelector('span').textContent = countdown.text

        // 更新紧迫样式
        IF countdown.isUrgent:
            countdownEl.classList.add('urgent')
        ELSE:
            countdownEl.classList.remove('urgent')

        // 自动标记为已结束

            card = DOM.querySelector("[data-id='" + gb.id + "']")
            IF card:
                card.classList.add('ended')
                ADD "已结束" badge TO card
            END IF
        END IF
    END FOR
END FUNCTION

// 调度入口（初始化时启动）

function getCountdown(endTime) {
    const now = new Date().getTime();
    const end = new Date(endTime).getTime();
    const diff = end - now;

    if (diff <= 0) {
        return { text: '已结束', isUrgent: false, isEnded: true };
    }

    const hours = Math.floor(diff / (1000 * 60 * 60));
    const minutes = Math.floor((diff % (1000 * 60 * 60)) / (1000 * 60));
    const seconds = Math.floor((diff % (1000 * 60)) / 1000);

    if (hours > 0) {
        return { text: `剩余 ${hours}小时${minutes}分`, isUrgent: false, isEnded: false };
    } else if (minutes > 0) {
        return { text: `剩余 ${minutes}分${seconds}秒`, isUrgent: true, isEnded: false };
    } else {
        return { text: `剩余 ${seconds}秒`, isUrgent: true, isEnded: false };
    }
}

// app.js 第 798-838 行 — 列表页批量更新
function updateListCountdowns() {

#### 2.5 算法分析

| 维度 | 说明 |
|------|------|
| **时间复杂度** | O(n)，n 为当前显示的拼单数量 |
| **空间复杂度** | O(1)，原地更新 DOM |

### 3. 个性化推荐算法

#### 3.1 算法概述

基于用户浏览历史（校区 + 商品类型偏好），从所有活跃拼单中筛选出最相关的拼单进行推荐。

#### 3.2 流程图

```
┌──────────────────────┐
│   用户打开应用         │
└──────────┬───────────┘
           ▼
┌──────────────────────────┐
│ 获取用户历史数据           │
│ getUserData().history    │
│ [{campus, category, count}]│
└──────────┬───────────────┘
           ▼
┌──────────────────────────┐
│ 获取所有活跃拼单           │
│ fetchGroupBuysFromAPI()  │
│ status === 'active'      │
└──────────┬───────────────┘
           ▼
    ┌──────────────┐
    │ 有无历史记录? │
    └──┬───────┬───┘
  有   │       │ 无（新用户）
       ▼       ▼
┌──────────┐  ┌──────────────┐
│提取高频   │  │直接取前4条    │
│校区和品类 │  │活跃拼单       │
│count排序 │  └──────┬───────┘
└────┬─────┘         │
     ▼               │
┌──────────────┐     │
│ 兴趣分数计算: │     │
│ 校区匹配+30分 │     │
│ 品类匹配+20分 │     │
│ 人数缺口-分数 │     │
│ 总分排序     │     │
└────┬─────────┘     │
     │               │
     └───────┬───────┘
             ▼
    ┌────────────────┐
    │ 取前 4 条       │
    │ (最多4个推荐)   │
    └───────┬────────┘
            ▼
    ┌────────────────┐
    │ 渲染推荐卡片UI  │
    │ renderRecommendation│
    │ Card()         │
    └────────────────┘
```

#### 3.3 伪代码

```
CONSTANT MAX_RECOMMENDATIONS = 4

FUNCTION getRecommendations():
    userData = getUserData()
    history = userData.history    // [{campus, category, count}, ...]

    groupBuys = fetchActiveGroupBuys()   // 从 API 获取所有 status=active 的拼单

    IF groupBuys.length == 0:
        RETURN []
    END IF

    // 情况1：无历史记录（新用户）
    IF history.length == 0:
        RETURN groupBuys.slice(0, MAX_RECOMMENDATIONS)
    END IF

    // 情况2：有历史，按兴趣评分排序
    // 找出用户最常访问的校区和品类
    favoriteCampus = history.sort(desc by count)[0].campus
    favoriteCategory = history.sort(desc by count)[0].category

    scored = []
    FOR EACH gb IN groupBuys:
        score = 0

        // 校区匹配：同校区 +30
        IF gb.campus == favoriteCampus:
            score = score + 30

        // 品类匹配：同品类 +20
        IF gb.category == favoriteCategory:
            score = score + 20

        // 参与进度衰减：人数快满了加分
        remainingSlots = gb.targetCount - gb.currentCount
        IF remainingSlots == 1:
            score = score + 10    // 仅剩1个名额，急迫加分

        scored.push({ groupBuy: gb, score: score })

    // 按分数降序排序
    scored.sort(desc by score)

    // 取前 N 个
    recommendations = []
    FOR i = 0 TO min(MAX_RECOMMENDATIONS, scored.length):
        recommendations.push(scored[i].groupBuy)

    RETURN recommendations
END FUNCTION

FUNCTION renderRecommendations():
    recommendations = getRecommendations()
    IF recommendations.length == 0:
        HIDE recommendationSection
        RETURN
    END IF

    recommendationSection.style.display = 'block'
    recommendationCount.textContent = recommendations.length + "个推荐"
    recommendationList.innerHTML = ''

    FOR EACH rb in recommendations:
        card = renderRecommendationCard(rb)
        recommendationList.appendChild(card)
END FUNCTION
```

// app.js 第 356-375 行 — 用户历史更新
function updateUserHistory(campus, category) {
    const userData = getUserData();
    const existingIndex = userData.history.findIndex(
        h => h.campus === campus && h.category === category
    );
    if (existingIndex !== -1) {
        userData.history[existingIndex].count = (userData.history[existingIndex].count || 1) + 1;
    } else {
        userData.history.push({ campus, category, count: 1 });
    }
    if (userData.history.length > 50) {
        userData.history = userData.history.slice(-50);
    }
    localStorage.setItem(CONFIG.USER_KEY, JSON.stringify(userData));
}

// app.js 第 581-631 行 — 推荐列表渲染（简化版取前4条）
async function renderRecommendations() {
    const recommendationSection = document.getElementById('recommendationSection');
    const recommendationList = document.getElementById('recommendationList');
    const recommendationCount = document.getElementById('recommendationCount');

    let groupBuys = [];
    try {
        if (CONFIG.USE_API) {
            const result = await apiCall('/groups');
            const rawData = result.data || [];
            groupBuys = rawData
                .filter(item => item.status === 'active')
                .map(item => ({ /* 数据格式转换 */ }));
        }
    } catch (error) { /* 错误处理 */ }

    if (groupBuys.length === 0) {
        recommendationSection.style.display = 'none';
        return;
    }

    const recommendations = groupBuys.slice(0, 4);
    recommendationSection.style.display = 'block';
    recommendationCount.textContent = `${recommendations.length}个推荐`;
    recommendationList.innerHTML = '';
    recommendations.forEach((rb, index) => {
        recommendationList.appendChild(renderRecommendationCard(rb, index));
    });
}
```

#### 3.5 算法扩展说明

当前实现采用**简化版推荐**（直接取前 4 条活跃拼单）。实际场景中可扩展为**基于用户兴趣评分**的推荐，评分规则如下：

| 匹配维度 | 权重 | 说明 |
|----------|------|------|
| 校区匹配 | +30 | 同校区/宿舍楼，物理距离近 |
| 品类匹配 | +20 | 历史高频品类，用户偏好 |
| 急迫加分 | +10 | 仅剩 1 个名额，促进成交 |
| 剩余名额 | +5 | 2 个名额也加分 |

这本质上是一个**基于内容（Content-Based）的轻量级推荐引擎**。

## 第二部分：设计模式应用说明

### 设计模式 1：观察者模式（Observer Pattern）

#### 1.1 应用场景

**倒计时实时更新系统**。系统每秒钟需要更新所有拼单卡片的剩余时间显示，当倒计时结束时自动标记拼单为"已结束"状态。

#### 1.2 问题分析

- 拼单卡片数量动态变化，需要批量更新所有卡片的倒计时文本
- 不同位置（列表页和推荐区）的倒计时需要同步更新
- 倒计时归零时需要触发状态变更（active → ended）

**不使用观察者模式**的痛点：
- 为每个卡片单独创建 `setInterval` 会大量消耗资源
- 卡片增删时需要手动管理定时器
- 难以统一维护倒计时逻辑

#### 1.3 UML 类图

```

```javascript
// ===== Subject（主题）：全局定时器 =====
// app.js 第 1345 行
setInterval(updateListCountdowns, 1000);   // 启动周期性通知

// ===== Observer（观察者）：各类倒计时显示 =====
// app.js 第 798-838 行
function updateListCountdowns() {
    const groupBuys = getGroupBuys();
    const listContainer = document.getElementById('groupbuyList');
    if (!listContainer) return;

    // 通知每个拼单卡片的倒计时 DOM Observer
    groupBuys.forEach(gb => {
        const countdownEl = document.getElementById(`countdown-${gb.id}`);
        if (!countdownEl) return;

        const countdown = getCountdown(gb.deadline);
        const spanEl = countdownEl.querySelector('span');
        if (spanEl) spanEl.textContent = countdown.text;

        if (countdown.isUrgent) {
            countdownEl.classList.add('urgent');
        } else {
            countdownEl.classList.remove('urgent');
        }

        // 倒计时结束 → 触发状态迁移（副作用链）
        if (countdown.isEnded && gb.status === 'active') {
            const card = listContainer.querySelector(`[data-id="${gb.id}"]`);
            if (card) {
                card.classList.add('ended');
                const badges = card.querySelector('.card-badges');
                if (badges && !badges.querySelector('.status-badge.ended')) {
                    const endedBadge = document.createElement('span');
                    endedBadge.className = 'status-badge ended';
                    endedBadge.textContent = '已结束';
                    badges.appendChild(endedBadge);
                }
            }
        }
    });

    // 级联通知推荐区的 Observer
    updateRecommendationCountdowns();
}

// 详情页独立的 Observer（递归 + requestAnimationFrame 调度）
// app.js 第 776-793 行
function startDetailCountdown(deadline) {
    const countdownEl = document.getElementById('detailCountdown');
    if (!countdownEl) return;

    const update = () => {
        const countdown = getCountdown(deadline);
        countdownEl.textContent = countdown.text;
        countdownEl.className = `deadline-countdown ${countdown.isEnded ? 'ended' : ''}`;

        if (!countdown.isEnded) {
            requestAnimationFrame(() => {
                setTimeout(update, 1000);
            });
        }
    };

    update();
}
```

#### 1.5 效果评价

| 维度 | 说明 |
|------|------|
| **解耦** | 倒计时逻辑与 UI 渲染分离，`getCountdown()` 纯函数只负责计算状态 |
| **性能** | 1 个全局定时器替代 N 个独立定时器，内存和 CPU 消耗降低 N 倍 |
| **可扩展** | 新增倒计时区域只需在 `updateListCountdowns()` 中添加一行调用 |
| **一致性** | 所有倒计时在同一时刻更新，不会出现不同卡片时间差的问题 |

**解决的问题**：避免了为每个卡片单独创建定时器的资源浪费，实现了统一的倒计时生命周期管理。

### 设计模式 2：工厂模式（Factory Pattern）

#### 2.1 应用场景

**拼单卡片 DOM 元素创建**。系统需要根据相同的拼单数据结构，在不同场景下生成不同样式的卡片：
- 列表页的标准卡片（`renderGroupBuyCard`）
- 推荐区的精简卡片（`renderRecommendationCard`）

#### 2.2 问题分析

- 两种卡片有**共享的渲染逻辑**（AA 计算、分类图标、校区名称、倒计时）
- 但布局结构和展示信息**存在差异**（推荐卡片更精简）
- 需要在多个地方调用，避免重复代码

**不使用工厂模式**的痛点：
- 每个调用点都需要完整实现卡片构建逻辑
- 卡片结构变更时需要修改多处代码
- 不同卡片类型之间缺乏统一的创建接口

### 设计模式 3：适配器模式（Adapter Pattern）

#### 3.1 应用场景

**前后端数据格式转换**。后端 Supabase 数据库使用蛇形命名（`snake_case`），字段名如 `title`、`total_amount`、`current_count`；前端 JavaScript 使用驼峰命名（`camelCase`），字段名如 `merchant`、`totalAmount`、`currentCount`。适配器负责双向转换。

#### 3.2 问题分析

- 后端数据库表 `groups` 的字段命名与前端模型不一致
- 分类值也不同：后端存中文（"外卖"），前端用英文键（"food"）
- 不做转换会导致代码中出现大量散乱的手动映射

**不使用适配器模式**的痛点：
- 每个使用后端数据的地方都要写一遍字段映射
- 字段变更时需修改多处代码
- 前后端耦合严重

#### 3.3 UML 类图

### 为什么选择这些模式？

1. **观察者模式** — 本项目核心是一个实时更新的列表应用（类似看板），倒计时是高频更新需求。观察者（轮询）模式是单页面应用中处理定时更新的最自然选择，用一个全局定时器统一调度所有 DOM 更新，相比为每个卡片独立创建定时器，性能和可维护性都有显著提升。

2. **工厂模式** — 列表卡片和推荐卡片本质上是从同一数据模型产出的不同"视图"。工厂模式将创建逻辑封装，使得"如何构建一张卡片"的知识内聚在函数内部，调用方只需要传入数据，不需要关心内部 HTML 结构。当需要新增"搜索结果的紧凑卡片"时，只需要新增一个工厂函数。

3. **适配器模式** — 后端 Supabase 使用 snake_case 和中文分类值（符合数据库约定），前端使用 camelCase 和英文枚举键（符合 JavaScript 约定）。适配器模式是处理这种命名风格差异的标准做法，避免了在每次 API 调用中散落大量手动映射代码。
